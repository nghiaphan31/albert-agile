## Semaine 4–5 : sprint_complete, CI 2 (develop→main) failure

### Z1 — sprint_number incrémenté avant CI 2 (develop→main) : désynchronisation en cas d'échec

La section III.12 (U6) dit : sprint_complete incrémente `state.sprint_number`. La section III.16 (Y3) dit : sprint_complete déclenche le nœud merge_to_main, et en cas d'échec CI 2 (develop→main) → H5 (approbation escalade API payante) avec `reason="ci2_github_actions_failure"` → R-4 (Albert Dev Team) corrige.

Problème : sprint_number est incrémenté (N→N+1) dès le début de sprint_complete, *avant* que merge_to_main soit exécuté. Si CI 2 (develop→main) échoue → H5 (approbation escalade API payante) → R-4 (Albert Dev Team) corrige → nouveau cycle E5 (tests CI) → sprint_complete se ré-exécute potentiellement avec sprint_number déjà à N+1.

**Correction** : Séparer sprint_complete en deux phases :  
1. `sprint_summary` (phase 1) : écriture sprint_summary dans BaseStore (mémoire long terme), index_rag, **sans incrémenter sprint_number**.  
2. `merge_to_main` (phase 2, après CI 2 (develop→main) vert) : **alors** incrémenter sprint_number et écrire dans BaseStore (mémoire long terme) sprint_counter.  
sprint_number = N tant que merge_to_main n'a pas réussi.

## Semaine 5–6 : Branch protection GitHub

### Z2 — gh pr merge bloqué par branch protection

Y3 dit : R-5 (Albert Release Manager) exécute `gh pr merge --merge --delete-branch=false` après CI 2 (develop→main) vert. Mais sur GitHub, `develop` et `main` peuvent avoir des règles de protection (required reviews, required status checks). `gh pr merge` échoue si ces règles ne sont pas satisfaites (ex. PR non approuvée par un reviewer humain).

**Manque** : Deux configurations supportées :  
- **Sans branch protection** (dépôt personnel/privé) : `gh pr merge` fonctionne directement. Documenté dans le README.  
- **Avec branch protection** (équipe, dépôt org) : `gh pr merge` échoue → interrupt H4 (Sprint Review)' (nouveau : validation manuelle merge). L'opérateur approuve la PR sur GitHub, puis `Command(resume="merged")`. Ou : désactiver la protection sur develop (seule main est protégée).  
Recommandation dans le spec : pour simplifier l'automatisation, ne pas activer de branch protection sur `develop`. Seule `main` peut avoir une protection (require CI vert seulement, pas de required reviewer).
