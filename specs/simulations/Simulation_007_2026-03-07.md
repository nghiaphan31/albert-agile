---

## Semaine 1–2 : Phase 0 + E2

### V1 — H5 sémantiquement impropre pour "3 cycles rejected H1/H2/H3"

U2 dit : « Si la limite de 3 cycles rejected est atteinte pour H1/H2/H3 → interrupt H5 (escalade humaine). »

H5 est défini comme « escalade N2 : approbation coût Opus/Sonnet ». Si R-0 et R-1 rejettent 3 fois le Gros Ticket, l'escalade est une impasse métier — pas un problème de coût. Utiliser H5 crée une confusion sémantique :
- L'opérateur voit "approbation N2" alors que le vrai problème est l'alignement sur la vision produit.
- Le payload H5 ne porterait pas le bon contexte (3 rejections successives vs escalade LLM).

**Manque** : Un mécanisme de "blocage critique" distinct de H5, ou la définition d'un nouveau type d'interrupt H7 (impasse sur vision/backlog/architecture). Ou : préciser que H5 est utilisé avec un payload `reason="max_rejections_H1"` pour distinguer le cas de la vraie escalade N2.

---

## Semaine 2–3 : Initialisation graphe multi-session

### V2 — Nœud d'initialisation absent : chargement BaseStore non décrit

U5 et U6 spécifient : `adr_counter` et `sprint_number` sont chargés depuis BaseStore au démarrage d'un nouveau thread. Mais :
- Aucun nœud "init_state" ou logique de chargement n'est décrit dans le graphe.
- Le TypedDict a des valeurs par défaut (adr_counter=0, sprint_number=1). Si BaseStore contient adr_counter=3 et qu'on démarre un nouveau thread sans charger le BaseStore, le compteur repart à 0 → collisions ADR.
- Qui déclenche ce chargement ? Un nœud "load_context" avant E1 ? Un `__init__` du graphe ?

**Manque** : Décrire un nœud "load_context" (ou hook de config LangGraph) exécuté à chaque démarrage de thread, qui lit les valeurs persistées (adr_counter, sprint_number, DoD) depuis le BaseStore et les injecte dans l'état initial.

---

## Semaine 3–4 : E2 et sprint N+1

### V3 — E2 déclenchement : une seule fois ou par sprint ?

Le flux V.5.2 est : E1→E2→E3→E4→E5→E6→(loop E3). E2 (Architecture + DoD) semble n'être exécuté qu'une seule fois par projet. Mais après 2 sprints, l'architecture a évolué :
- Sprint 1 : CLI core commands → Architecture v1.
- Sprint 2 : ajout module plugins → R-2 doit amender Architecture.md.
- L'architecture change → la DoD évolue aussi.

**Manque** : Définir si E2 est déclenché uniquement à l'initialisation (Phase 0 + E1) ou si un nœud "E2 review" peut être re-déclenché entre sprints. Option : si le Sprint Backlog de H3 contient des tickets "architectural change", sprint_complete boucle vers E2 au lieu de E3.

---

## Semaine 5–8 : E4 feature branches et stratégie Git

### V4 — Stratégie de branches Git non définie

Le spec dit que R-5 gère les branches Git, mais la topologie n'est jamais spécifiée :
- La feature branch est créée à partir de quelle branche ? `main` ? `develop` ? La spec utilise "feature branch" (E4) et "merge final develop→main" (U8). Ces deux branches (develop, main) sont implicites mais jamais définies.
- À chaque ticket dans E4 : R-5 crée une feature branch par ticket ? Ou une feature branch par sprint ? Si par ticket, le merge dans develop (ou main) se fait quand ? Après E5 (CI vert) ou après H4 (Go/No-Go) ?
- GitHub Actions (U7) valide les PRs "avant merge main". Mais si feature → develop d'abord, GitHub Actions valide sur develop→main ? Ou sur feature→main directement ?

**Manque** : Documenter la stratégie de branches : branche par sprint, merge dans develop après E5, puis develop→main après H4 (Go/No-Go). Ou : trunk-based development (feature → main direct). Choisir et documenter.

---

## Semaine 8–11 : E5 H4 et H5 dans le même nœud

### V5 — Ordre de priorité H4 vs H5 dans E5

E5 peut déclencher deux interrupts : H5 (escalade N2 pendant Self-Healing) ET H4 (CI vert, fin E5). Si le Self-Healing atteint 3 cycles (H5) mais que CI passe quand même après l'escalade, est-ce que H4 est déclenché après H5 ? Ou H5 annule H4 ?

Séquence ambiguë : (1) R-6 détecte échec tests → Self-Healing R-6→R-4. (2) Cycle 3 : H5 (escalade N2). (3) R-1 approuve N2. R-4 corrige via Opus/Sonnet. (4) CI passe. → H4 doit être déclenché. Mais le graphe a déjà traversé H5. Le nœud E5 doit savoir qu'il faut déclencher H4 après H5 résolu. Cette séquence n'est pas décrite.

**Manque** : Préciser que H4 est toujours déclenché quand CI passe, indépendamment du nombre de H5 qui ont précédé. H5 est une interruption intra-E5 (bloquante mais temporaire). H4 = sortie de E5 vers E6.

---

## Semaine 11–13 : BaseStore écriture DoD et sprint N+1

### V6 — DoD immuable ou versionnée par sprint ?

U1 stocke la DoD dans `state.dod` et BaseStore (`project/{id}/dod`). Mais lors du sprint 2 :
- Si la DoD change (V3 : R-2 amende Architecture.md), le BaseStore écrase-t-il la DoD précédente ?
- R-6 utilise la DoD de quel sprint en E5 ?
- En cas d'audit (DoD sprint 1 vs DoD sprint 2), est-ce possible de retrouver les versions ?

**Manque** : Versionner la DoD dans le BaseStore : namespace `project/{id}/dod/{sprint_number}`. load_context charge la dernière version. Lors d'une mise à jour (H2 d'un sprint N+1 si E2 est re-déclenché), écrire dans `project/{id}/dod/{sprint_number}` sans écraser les précédentes.

---

## Semaine 13–16 : Post-H4 rejected et feature branch

### V7 — H4 rejected : état de la feature branch

U2 dit : « H4 rejected → R-4/R-5 traitent le correctif (nouveau cycle E4→E5). » Mais :
- La feature branch du sprint est déjà complète (tous les tickets). Si H4 rejected, R-4 génère des commits correctifs. Ces commits vont-ils sur la même feature branch ? Ou une nouvelle branche ?
- Si même branche : l'historique Git devient confus (mélange développement sprint + corrections post-review).
- Si nouvelle branche : comment R-5 gère le merge avec la première ?
- GitHub Actions (U7) : H4 rejected après CI vert local (E5). Le CI local a passé. La correction post-H4 doit re-passer CI local (R-6) et CI GitHub Actions. Ce sous-cycle n'est pas décrit.

**Manque** : Spécifier qu'un H4 rejected crée une branche correctrice (`{project_id}-sprint-{NN}-fix-{attempt}`) ou des commits supplémentaires sur la même feature branch. Puis : nouveau cycle E5 (CI local) et GitHub Actions avant de soumettre à H4 à nouveau.

## Conclusion Simulation 007

Sept problèmes résiduels identifiés (V1–V7), dont trois majeurs. La correction de U1–U8 a généré des nouvelles ambiguïtés sur l'initialisation du graphe, la topologie Git, et la coexistence de H4/H5 dans E5. Corrections intégrées dans le spec, section III.13.
