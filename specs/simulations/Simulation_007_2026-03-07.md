## Semaine 1–2 : Phase 0 + E2 (architecture)

### V1 — H5 (approbation escalade API payante) sémantiquement impropre pour "3 cycles rejected H1 (validation Gros Ticket)/H2 (validation Architecture + DoD) (validation Architecture + DoD (Definition of Done))/H3 (validation Sprint Backlog)"

U2 dit : « Si la limite de 3 cycles rejected est atteinte pour H1 (validation Gros Ticket)/H2 (validation Architecture + DoD) (validation Architecture + DoD (Definition of Done))/H3 (validation Sprint Backlog) → interrupt H5 (approbation escalade API payante) (escalade humaine). »

H5 (approbation escalade API payante) est défini comme « escalade N2 (cloud payant) : approbation coût Opus/Sonnet ». Si R-0 (Albert Business Analyst) et R-1 (Nghia Product Owner) rejettent 3 fois le Gros Ticket, l'escalade est une impasse métier — pas un problème de coût. Utiliser H5 (approbation escalade API payante) crée une confusion sémantique :
- L'opérateur voit "approbation N2 (cloud payant)" alors que le vrai problème est l'alignement sur la vision produit.
- Le payload H5 (approbation escalade API payante) ne porterait pas le bon contexte (3 rejections successives vs escalade LLM).

**Manque** : Un mécanisme de "blocage critique" distinct de H5 (approbation escalade API payante), ou la définition d'un nouveau type d'interrupt H7 (impasse sur vision/backlog/architecture). Ou : préciser que H5 (approbation escalade API payante) est utilisé avec un payload `reason="max_rejections_H1 (validation Gros Ticket)"` pour distinguer le cas de la vraie escalade N2 (cloud payant).

## Semaine 2–3 : Initialisation graphe multi-session

### V2 — Nœud d'initialisation absent : chargement BaseStore (mémoire long terme) non décrit

U5 et U6 spécifient : `adr_counter` et `sprint_number` sont chargés depuis BaseStore (mémoire long terme) au démarrage d'un nouveau thread. Mais :
- Aucun nœud "init_state" ou logique de chargement n'est décrit dans le graphe.
- Le TypedDict a des valeurs par défaut (adr_counter=0, sprint_number=1). Si BaseStore (mémoire long terme) contient adr_counter=3 et qu'on démarre un nouveau thread sans charger le BaseStore (mémoire long terme), le compteur repart à 0 → collisions ADR (Architecture Decision Record).
- Qui déclenche ce chargement ? Un nœud "load_context" avant E1 (idéation) ? Un `__init__` du graphe ?

**Manque** : Décrire un nœud "load_context" (ou hook de config LangGraph) exécuté à chaque démarrage de thread, qui lit les valeurs persistées (adr_counter, sprint_number, DoD (Definition of Done)) depuis le BaseStore (mémoire long terme) et les injecte dans l'état initial.

## Semaine 3–4 : E2 (architecture) et sprint N+1

### V3 — E2 (architecture) déclenchement : une seule fois ou par sprint ?

Le flux V.5.2 est : E1 (idéation)→E2 (architecture)→E3 (Sprint Backlog)→E4 (exécution code)→E5 (tests CI)→E6 (clôture sprint)→(loop E3 (Sprint Backlog)). E2 (architecture) (Architecture + DoD (Definition of Done)) semble n'être exécuté qu'une seule fois par projet. Mais après 2 sprints, l'architecture a évolué :
- Sprint 1 : CLI core commands → Architecture v1.
- Sprint 2 : ajout module plugins → R-2 (Albert System Architect) doit amender Architecture.md.
- L'architecture change → la DoD (Definition of Done) évolue aussi.

**Manque** : Définir si E2 (architecture) est déclenché uniquement à l'initialisation (Phase 0 + E1 (idéation)) ou si un nœud "E2 (architecture) review" peut être re-déclenché entre sprints. Option : si le Sprint Backlog de H3 (validation Sprint Backlog) contient des tickets "architectural change", sprint_complete boucle vers E2 (architecture) au lieu de E3 (Sprint Backlog).

## Semaine 5–8 : E4 (exécution code) feature branches et stratégie Git

### V4 — Stratégie de branches Git non définie

Le spec dit que R-5 (Albert Release Manager) gère les branches Git, mais la topologie n'est jamais spécifiée :
- La feature branch est créée à partir de quelle branche ? `main` ? `develop` ? La spec utilise "feature branch" (E4 (exécution code)) et "merge final develop→main" (U8). Ces deux branches (develop, main) sont implicites mais jamais définies.
- À chaque ticket dans E4 (exécution code) : R-5 (Albert Release Manager) crée une feature branch par ticket ? Ou une feature branch par sprint ? Si par ticket, le merge dans develop (ou main) se fait quand ? Après E5 (tests CI) (CI vert) ou après H4 (Sprint Review) (Go/No-Go) ?
- GitHub Actions (U7) valide les PRs "avant merge main". Mais si feature → develop d'abord, GitHub Actions valide sur develop→main ? Ou sur feature→main directement ?

**Manque** : Documenter la stratégie de branches : branche par sprint, merge dans develop après E5 (tests CI), puis develop→main après H4 (Sprint Review) (Go/No-Go). Ou : trunk-based development (feature → main direct). Choisir et documenter.

## Semaine 8–11 : E5 (tests CI) H4 (Sprint Review) et H5 (approbation escalade API payante) dans le même nœud

### V5 — Ordre de priorité H4 (Sprint Review) vs H5 (approbation escalade API payante) dans E5 (tests CI)

E5 (tests CI) peut déclencher deux interrupts : H5 (approbation escalade API payante) (escalade N2 (cloud payant) pendant Self-Healing) ET H4 (Sprint Review) (CI vert, fin E5 (tests CI)). Si le Self-Healing atteint 3 cycles (H5 (approbation escalade API payante)) mais que CI passe quand même après l'escalade, est-ce que H4 (Sprint Review) est déclenché après H5 (approbation escalade API payante) ? Ou H5 (approbation escalade API payante) annule H4 (Sprint Review) ?

Séquence ambiguë : (1) R-6 (Albert QA & DevOps) détecte échec tests → Self-Healing R-6 (Albert QA & DevOps)→R-4 (Albert Dev Team). (2) Cycle 3 : H5 (approbation escalade API payante) (escalade N2 (cloud payant)). (3) R-1 (Nghia Product Owner) approuve N2 (cloud payant). R-4 (Albert Dev Team) corrige via Opus/Sonnet. (4) CI passe. → H4 (Sprint Review) doit être déclenché. Mais le graphe a déjà traversé H5 (approbation escalade API payante). Le nœud E5 (tests CI) doit savoir qu'il faut déclencher H4 (Sprint Review) après H5 (approbation escalade API payante) résolu. Cette séquence n'est pas décrite.

**Manque** : Préciser que H4 (Sprint Review) est toujours déclenché quand CI passe, indépendamment du nombre de H5 (approbation escalade API payante) qui ont précédé. H5 (approbation escalade API payante) est une interruption intra-E5 (tests CI) (bloquante mais temporaire). H4 (Sprint Review) = sortie de E5 (tests CI) vers E6 (clôture sprint).

## Semaine 11–13 : BaseStore (mémoire long terme) écriture DoD (Definition of Done) et sprint N+1

### V6 — DoD (Definition of Done) immuable ou versionnée par sprint ?

U1 stocke la DoD (Definition of Done) dans `state.dod` et BaseStore (mémoire long terme) (`project/{id}/dod`). Mais lors du sprint 2 :
- Si la DoD (Definition of Done) change (V3 : R-2 (Albert System Architect) amende Architecture.md), le BaseStore (mémoire long terme) écrase-t-il la DoD (Definition of Done) précédente ?
- R-6 (Albert QA & DevOps) utilise la DoD (Definition of Done) de quel sprint en E5 (tests CI) ?
- En cas d'audit (DoD (Definition of Done) sprint 1 vs DoD (Definition of Done) sprint 2), est-ce possible de retrouver les versions ?

**Manque** : Versionner la DoD (Definition of Done) dans le BaseStore (mémoire long terme) : namespace `project/{id}/dod/{sprint_number}`. load_context charge la dernière version. Lors d'une mise à jour (H2 (validation Architecture + DoD) (validation Architecture + DoD (Definition of Done)) d'un sprint N+1 si E2 (architecture) est re-déclenché), écrire dans `project/{id}/dod/{sprint_number}` sans écraser les précédentes.

## Semaine 13–16 : Post-H4 (Sprint Review) rejected et feature branch

### V7 — H4 (Sprint Review) rejected : état de la feature branch

U2 dit : « H4 (Sprint Review) rejected → R-4 (Albert Dev Team)/R-5 (Albert Release Manager) traitent le correctif (nouveau cycle E4 (exécution code)→E5 (tests CI)). » Mais :
- La feature branch du sprint est déjà complète (tous les tickets). Si H4 (Sprint Review) rejected, R-4 (Albert Dev Team) génère des commits correctifs. Ces commits vont-ils sur la même feature branch ? Ou une nouvelle branche ?
- Si même branche : l'historique Git devient confus (mélange développement sprint + corrections post-review).
- Si nouvelle branche : comment R-5 (Albert Release Manager) gère le merge avec la première ?
- GitHub Actions (U7) : H4 (Sprint Review) rejected après CI vert local (E5 (tests CI)). Le CI local a passé. La correction post-H4 (Sprint Review) doit re-passer CI local (R-6 (Albert QA & DevOps)) et CI GitHub Actions. Ce sous-cycle n'est pas décrit.

**Manque** : Spécifier qu'un H4 (Sprint Review) rejected crée une branche correctrice (`{project_id}-sprint-{NN}-fix-{attempt}`) ou des commits supplémentaires sur la même feature branch. Puis : nouveau cycle E5 (tests CI) (CI local) et GitHub Actions avant de soumettre à H4 (Sprint Review) à nouveau.
