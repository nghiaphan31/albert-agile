## Semaine 1–2 : Démarrage projet, load_context

### W1 — Entry point du graphe : E1 vs E3 non différenciés

V2 dit : « Nœud load_context exécuté avant E1 (nouveau projet) ou avant E3 (sprint N+1). » Mais le graphe LangGraph a un unique point d'entrée (le nœud START). Comment le graphe sait-il s'il démarre un nouveau projet (→ E1) ou un nouveau sprint (→ E3) ?

Options non définies dans le spec :
- Option A : deux graphes distincts (un par mode). Complexité accrue.
- Option B : un seul graphe avec un paramètre de config `start_phase: "E1" | "E3"` passé dans `configurable`. Le nœud load_context route selon ce paramètre.
- Option C : load_context détecte si un Backlog et Architecture.md existent dans l'état/BaseStore. Si non → E1. Si oui → E3.

**Manque** : La spec ne précise pas comment l'opérateur invoque le graphe pour un nouveau sprint (vs nouveau projet). Exemple CLI manquant : `python run_graph.py --project-id cli --phase E3 --thread-id cli-sprint-02`.

## Semaine 4–6 : Sprint 2 + needs_architecture_review

### W2 — Qui positionne needs_architecture_review et quand ?

V3 dit : « sprint_complete positionne `needs_architecture_review=True` si le Sprint Backlog contient des tickets taggués `#architectural-change` ou si R-3 l'a signalé. »

Ambiguïté :
- Qui tag les tickets `#architectural-change` ? R-1 humain lors de la création du Backlog ? R-3 lors du découpage (E3) ? Le tag n'est pas un artefact structuré du Sprint Backlog Pydantic.
- « Si R-3 l'a signalé » : R-3 retourne-t-il un champ `architectural_change: bool` dans son output Pydantic ? Ce champ n'est pas défini dans le schéma Sprint Backlog.

**Manque** : (1) Ajouter `architectural_change: bool` dans le schéma Pydantic du Sprint Backlog (sortie R-3). Si True : sprint_complete met `needs_architecture_review=True`. (2) R-3 détermine `architectural_change` en comparant les nouveaux tickets à l'Architecture.md existant via RAG.

## Semaine 6–8 : Re-déclenchement E2 sprint 2

Sprint 2 détecte un changement architectural. sprint_complete boucle vers E2 (V3). H2 redéclenché. Tout se passe bien. Mais :

### W3 — BaseStore DoD manquante pour sprint_number=2 avant re-E2

Séquence : sprint_complete (sprint 1) → needs_architecture_review=True → boucle E2. Mais sprint_number a déjà été incrémenté à 2 (par sprint_complete V6, U6). Lors de la prochaine exécution de load_context (thread cli-sprint-02) : load_context cherche `project/cli/dod/2`. Cette DoD n'existe pas encore (le H2 de sprint 2 n'a pas encore eu lieu). Load_context renvoie `dod=None`.

Si E3 utilise `state.dod=None` par inadvertance (ex. R-3 consulte la DoD pour valider les critères d'acceptation) : crash ou comportement non défini.

**Manque** : Préciser que si `project/{id}/dod/{sprint_number}` est absent dans BaseStore, load_context charge la *dernière version disponible* (`project/{id}/dod/{sprint_number - 1}`). Jamais None si une DoD de sprint précédent existe. Si aucune DoD disponible (sprint 1, première exécution avant E2) : `dod=None` est attendu → E2 est obligatoire.

## Semaine 9–14 : Sprint 3 + GitHub Actions failure

### W4 — GitHub Actions failure après H4 approuvé : état graphe incohérent

V4 dit : « Après H4 approuvé, R-5 merge feature→develop. GitHub Actions déclenché sur PR develop→main. » U7 dit : « En cas de divergence R-6 passe / GitHub Actions échoue : priorité GitHub Actions ; R-4 corrige et relance. »

Sprint 3 : R-6 passe (CI local vert). H4 interrupt déclenché. R-7 approuve (Go/No-Go). `Command(resume="approved")`. Le graphe reprend en E6 (post-H4). sprint_complete est exécuté (sprint_summary BaseStore, sprint_number++). Puis R-5 merge feature→develop et ouvre la PR develop→main. GitHub Actions échoue.

Problème : Le graphe est **terminé** (sprint_complete a fini). Il n'y a plus d'état actif pour ce thread. Mais le merge est bloqué. Qui corrige ? Le spec dit "R-4 corrige et relance" mais le graphe est terminé.

**Manque** : Deux options à documenter : (A) R-5 pousse la feature branch avant H4 (pas après), GitHub Actions tourne sur la PR *avant* H4 → H4 n'est déclenché que si GitHub Actions est vert. Ou : (B) sprint_complete attend le retour GitHub Actions avant de se terminer (monitoring du status PR). Option A plus simple et robuste : H4 = CI local vert + CI GitHub Actions vert.

## Semaine 14–16 : Hotfix production

### W5 — Absence de workflow hotfix

V4 définit la stratégie feature-per-sprint + develop + main. Mais que se passe-t-il quand un bug critique est découvert en production (main) entre deux sprints ?

- Un hotfix ne correspond pas à un sprint.
- Le graphe est en attente (pas de thread actif).
- Qui crée la branche hotfix ? Comment est-elle intégrée ?

**Manque** : Définir un workflow hotfix minimal : branche `hotfix/{project_id}-{date}-{description}` depuis main. Correction manuelle ou via invocation `--phase E4` du graphe avec thread_id `{project_id}-hotfix-{date}`. Merge hotfix→main + hotfix→develop. R-5 applique.

---

## Semaine 16–18 : Fin projet

### W6 — projects.json : emplacement et accès runtime non définis

La checklist 4.1 mentionne `projects.json` comme fichier de configuration. Les nœuds du graphe (sprint_complete, load_context) lisent ce fichier. Mais :
- Son chemin absolu n'est pas défini dans le spec. Est-il à la racine du projet orchestration ? Dans un répertoire `config/` ?
- Comment les nœuds le trouvent à runtime ? Variable d'environnement `AGILE_PROJECTS_JSON=/path/projects.json` ? Chemin relatif à `AGILE_ORCHESTRATION_ROOT` ?
- Si l'opérateur déplace le projet orchestration : le chemin en dur dans les nœuds est faux.

**Manque** : Spécifier `AGILE_PROJECTS_JSON` comme variable d'environnement (défaut : `$AGILE_ORCHESTRATION_ROOT/config/projects.json`). Configurer dans `.agile-env` ou en variable système.
