## Semaine 1–2 : Phase 0 + E1 (idéation) + Co-construction DoD (Definition of Done)

### U1 — Co-construction de la DoD (Definition of Done) : aucun nœud ni interrupt défini

Le tableau V.5.2 dit : « R-2 (Albert System Architect) et R-1 (Nghia Product Owner) co-construisent la DoD (Definition of Done) » dans E2 (architecture). La DoD (Definition of Done) est un artefact critique (R-6 (Albert QA & DevOps) l'utilise en E5 (tests CI) comme grille de validation). Mais :
- R-1 (Nghia Product Owner) est humain, sans nœud dans le graphe.
- Aucun interrupt H1 (validation Gros Ticket)–H6 (résolution conflit Git) n'est prévu pour la co-construction de la DoD (Definition of Done).
- Le graphe n'a pas de nœud "dod_creation" ni de step intermédiaire dans E2 (architecture).
- Comment R-1 (Nghia Product Owner) contribue-t-il à la DoD (Definition of Done) dans le graphe ? En dehors du graphe (éditeur de texte) ? Via un interrupt H2 (validation Architecture + DoD) (validation Architecture + DoD (Definition of Done)) enrichi ?

**Manque** : Le flux de création de la DoD (Definition of Done) n'est pas décrit. Est-ce que H2 (validation Architecture + DoD) (validation Architecture + DoD (Definition of Done)) transmet Architecture.md *et* un draft de DoD (Definition of Done) ? R-1 (Nghia Product Owner) valide/amende les deux ? Le doc ne le précise pas.

## Semaine 4–6 : E4 (exécution code) sprint 1, RAG (recherche sémantique) hook et GPU

### U3 — Indexation post-commit (hook) pendant E4 (exécution code) : conflit GPU

III.8 dit : « Éviter d'exécuter des indexations RAG (recherche sémantique) massives pendant E4 (exécution code)/E5 (tests CI) ». Mais le hook Git post-commit s'exécute automatiquement à chaque commit, et E4 (exécution code) génère des commits à chaque ticket (R-5 (Albert Release Manager)). Donc pendant E4 (exécution code), le hook lance nomic-embed-text sur Ollama après chaque commit. nomic-embed-text occupe la même ressource que qwen2.5-coder (GPU). Le conseil "éviter les indexations pendant E4 (exécution code)" est en contradiction avec le hook automatique.

**Manque** : Aucun mécanisme décrit pour désactiver/différer le hook pendant E4 (exécution code). Option : variable `AGILE_DEFER_INDEX=true` dans `.agile-env` pour que le hook s'enregistre dans une file et indexe en fin de sprint plutôt qu'après chaque commit.

## Semaine 6–7 : E5 (tests CI) et H4 (Sprint Review)

### U4 — Ambiguïté H4 (Sprint Review) : trigger CI/CD au vert vs fin E6 (clôture sprint)

Le tableau III.6 dit : H4 (Sprint Review) déclenché par « Pipeline CI/CD au vert ». Le tableau V.5.2 dit : H4 (Sprint Review) = « R-7 (Nghia Stakeholder) valide l'incrément avant release ». Ces deux descriptions entrent en tension :
- Si H4 (Sprint Review) = CI/CD au vert (fin E5 (tests CI)) : le graphe interrompt après E5 (tests CI) pour que R-7 (Nghia Stakeholder) fasse le Go/No-Go. C'est logique.
- Si H4 (Sprint Review) = fin E6 (clôture sprint) : l'interrupt survient à la fin du Sprint Review, après que R-7 (Nghia Stakeholder) a déjà vu l'incrément. Mais alors qui déclenche le Sprint Review ?

**Manque** : Préciser si H4 (Sprint Review) est déclenché (a) juste après E5 (tests CI) (CI vert, avant Sprint Review) ou (b) à la fin du Sprint Review (E6 (clôture sprint)). Le tableau III.6 et V.5.2 sont incohérents sur ce point.

S11 (III.10) : « R-2 (Albert System Architect) produit un ADR (Architecture Decision Record)-0 (Business Analyst)01 à la fin de E2 (architecture) via template Pydantic. » Pour le projet dashboard, sprint 2 → R-2 (Albert System Architect) amende Architecture.md. Produit-il ADR (Architecture Decision Record)-0 (Business Analyst)01 (même nom) ou ADR (Architecture Decision Record)-0 (Business Analyst)02 ? Si ADR (Architecture Decision Record)-0 (Business Analyst)01 écrase le précédent : perte de traçabilité. Si ADR (Architecture Decision Record)-0 (Business Analyst)02 : qui gère le compteur ? L'état du graphe ? Le BaseStore (mémoire long terme) ?

**Manque** : Convention de numérotation des ADRs et emplacement du compteur. Recommandé : namespace BaseStore (mémoire long terme) `project/{id}/adr_counter` incrémenté par R-2 (Albert System Architect). Fichiers : `ADR (Architecture Decision Record)-0 (Business Analyst)01-<date>-<titre-court>.md`.

## Semaine 9–11 : sprint_complete et sprint_number

### U6 — Compteur de sprint dans auto_next_sprint

S4 (III.10) : Avec `auto_next_sprint=true`, le nœud sprint_complete génère `thread_id={project_id}-sprint-{NN}` (N+1). Mais où est stocké N (le numéro de sprint actuel) ? Le thread_id courant contient `sprint-01`. Pour dériver N+1=02 : le nœud doit parser le thread_id courant. Mais si le thread_id a un format non standard (reprise manuelle avec un nom libre) : le parsing échoue.

**Manque** : Stocker le sprint_number explicitement dans l'état TypedDict (`sprint_number: int`) et dans le BaseStore (mémoire long terme) (namespace `project/{id}/sprint_counter`). sprint_complete incrémente ce compteur. Plus fiable que le parsing du thread_id.

## Semaine 11–13 : GitHub Actions vs R-6 (Albert QA & DevOps)

### U7 — Relation GitHub Actions / nœud R-6 (Albert QA & DevOps)

Section IV et VII mentionnent GitHub Actions pour CI/CD. Section V.5.2 décrit R-6 (Albert QA & DevOps) comme exécutant les tests via Docker (LangGraph). Ces deux pipelines sont-ils alternatifs ou complémentaires ?
- Si alternatifs : un seul est utilisé. Lequel ?
- Si complémentaires : R-6 (Albert QA & DevOps) (LangGraph) lance les tests locaux ; GitHub Actions fait les tests sur PR/push. Mais R-6 (Albert QA & DevOps) agit avant le merge (feature branch) ; GitHub Actions après. Qui a le dernier mot sur la DoD (Definition of Done) ?

**Manque** : Le spec ne précise pas si GitHub Actions est le CI "officiel" de production ou juste un CI optionnel. Si R-6 (Albert QA & DevOps) (local, Docker) passe mais GitHub Actions (remote) échoue : incohérence. Recommandation : documenter la relation explicitement.
