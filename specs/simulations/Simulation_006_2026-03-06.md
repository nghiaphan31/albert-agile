## Semaine 1–2 : Phase 0 + E1 + Co-construction DoD

### U1 — Co-construction de la DoD : aucun nœud ni interrupt défini

Le tableau V.5.2 dit : « R-2 et R-1 co-construisent la DoD » dans E2. La DoD est un artefact critique (R-6 l'utilise en E5 comme grille de validation). Mais :
- R-1 est humain, sans nœud dans le graphe.
- Aucun interrupt H1–H6 n'est prévu pour la co-construction de la DoD.
- Le graphe n'a pas de nœud "dod_creation" ni de step intermédiaire dans E2.
- Comment R-1 contribue-t-il à la DoD dans le graphe ? En dehors du graphe (éditeur de texte) ? Via un interrupt H2 enrichi ?

**Manque** : Le flux de création de la DoD n'est pas décrit. Est-ce que H2 transmet Architecture.md *et* un draft de DoD ? R-1 valide/amende les deux ? Le doc ne le précise pas.

## Semaine 3–5 : E2 rejection + boucles H

### U2 — Comportement du graphe sur resume "rejected"

III.8 mentionne `resume={"status":"rejected","resume_after":"2026-03-05"}`. Mais :
- Pour H1 (rejected) : le graphe doit-il relancer R-0 pour un nouveau Gros Ticket ? Ou s'arrêter ?
- Pour H2 (rejected) : le graphe doit-il relancer R-2 avec le feedback de R-1/R-7 ?
- Pour H3 (rejected) : le graphe doit-il relancer R-3 avec le feedback ?

**Manque** : Les branches "rejected" après chaque interrupt H ne sont pas spécifiées. La section III.6 décrit uniquement le cas "approved". Un rejected sans boucle de correction = pipeline bloqué ou terminé en erreur.

## Semaine 4–6 : E4 sprint 1, RAG hook et GPU

### U3 — Indexation post-commit (hook) pendant E4 : conflit GPU

III.8 dit : « Éviter d'exécuter des indexations RAG massives pendant E4/E5 ». Mais le hook Git post-commit s'exécute automatiquement à chaque commit, et E4 génère des commits à chaque ticket (R-5). Donc pendant E4, le hook lance nomic-embed-text sur Ollama après chaque commit. nomic-embed-text occupe la même ressource que qwen2.5-coder (GPU). Le conseil "éviter les indexations pendant E4" est en contradiction avec le hook automatique.

**Manque** : Aucun mécanisme décrit pour désactiver/différer le hook pendant E4. Option : variable `AGILE_DEFER_INDEX=true` dans `.agile-env` pour que le hook s'enregistre dans une file et indexe en fin de sprint plutôt qu'après chaque commit.

## Semaine 6–7 : E5 et H4

### U4 — Ambiguïté H4 : trigger CI/CD au vert vs fin E6

Le tableau III.6 dit : H4 déclenché par « Pipeline CI/CD au vert ». Le tableau V.5.2 dit : H4 = « R-7 valide l'incrément avant release ». Ces deux descriptions entrent en tension :
- Si H4 = CI/CD au vert (fin E5) : le graphe interrompt après E5 pour que R-7 fasse le Go/No-Go. C'est logique.
- Si H4 = fin E6 : l'interrupt survient à la fin du Sprint Review, après que R-7 a déjà vu l'incrément. Mais alors qui déclenche le Sprint Review ?

**Manque** : Préciser si H4 est déclenché (a) juste après E5 (CI vert, avant Sprint Review) ou (b) à la fin du Sprint Review (E6). Le tableau III.6 et V.5.2 sont incohérents sur ce point.

## Semaine 7–9 : BaseStore et numérotation ADR

### U5 — Numérotation des ADRs

S11 (III.10) : « R-2 produit un ADR-001 à la fin de E2 via template Pydantic. » Pour le projet dashboard, sprint 2 → R-2 amende Architecture.md. Produit-il ADR-001 (même nom) ou ADR-002 ? Si ADR-001 écrase le précédent : perte de traçabilité. Si ADR-002 : qui gère le compteur ? L'état du graphe ? Le BaseStore ?

**Manque** : Convention de numérotation des ADRs et emplacement du compteur. Recommandé : namespace BaseStore `project/{id}/adr_counter` incrémenté par R-2. Fichiers : `ADR-001-<date>-<titre-court>.md`.

## Semaine 9–11 : sprint_complete et sprint_number

### U6 — Compteur de sprint dans auto_next_sprint

S4 (III.10) : Avec `auto_next_sprint=true`, le nœud sprint_complete génère `thread_id={project_id}-sprint-{NN}` (N+1). Mais où est stocké N (le numéro de sprint actuel) ? Le thread_id courant contient `sprint-01`. Pour dériver N+1=02 : le nœud doit parser le thread_id courant. Mais si le thread_id a un format non standard (reprise manuelle avec un nom libre) : le parsing échoue.

**Manque** : Stocker le sprint_number explicitement dans l'état TypedDict (`sprint_number: int`) et dans le BaseStore (namespace `project/{id}/sprint_counter`). sprint_complete incrémente ce compteur. Plus fiable que le parsing du thread_id.

## Semaine 11–13 : GitHub Actions vs R-6

### U7 — Relation GitHub Actions / nœud R-6

Section IV et VII mentionnent GitHub Actions pour CI/CD. Section V.5.2 décrit R-6 comme exécutant les tests via Docker (LangGraph). Ces deux pipelines sont-ils alternatifs ou complémentaires ?
- Si alternatifs : un seul est utilisé. Lequel ?
- Si complémentaires : R-6 (LangGraph) lance les tests locaux ; GitHub Actions fait les tests sur PR/push. Mais R-6 agit avant le merge (feature branch) ; GitHub Actions après. Qui a le dernier mot sur la DoD ?

**Manque** : Le spec ne précise pas si GitHub Actions est le CI "officiel" de production ou juste un CI optionnel. Si R-6 (local, Docker) passe mais GitHub Actions (remote) échoue : incohérence. Recommandation : documenter la relation explicitement.

## Semaine 13–14 : Fin de projet, archivage

### U8 — Procédure de fin de projet non décrite

Quand un projet est terminé (tous les tickets livrés, product stable) :
- Qui met `"archived": true` dans projects.json ? R-1/R-7 manuellement ? Un nœud "project_close" ?
- Les collections Chroma doivent être exportées (S10). Qui déclenche `export_chroma.py` ?
- Les checkpoints actifs du projet doivent être conservés (ou purgés selon politique).
- Le dépôt Git : merge de la branche de développement principale dans main ? R-5 fait-il ce merge final ?

**Manque** : Procédure de clôture de projet absente du spec. Aucun nœud "project_close" ni checklist de fin de projet.
