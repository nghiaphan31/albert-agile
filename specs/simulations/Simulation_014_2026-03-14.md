## Contraintes matérielles — RTX 3060 12 Go

| Composant | VRAM | Comportement |
|-----------|------|--------------|
| **qwen2.5-coder:7b** | ~8–10 Go | R-3, R-4, R-5, R-6 : un seul chargement à la fois |
| **gemma3:12b-it-q4_K_M** | ~8,1 Go | R-0, R-2 : swapping si qwen2.5 chargé |
| **nomic-embed-text** | ~1–2 Go (estimé) | Embeddings Chroma ; conflit si LLM actif |
| **Total disponible** | 12 Go | **Impossible** : qwen2.5 + gemma3 en parallèle |

**Conséquences** : Ollama charge un seul modèle à la fois. Switch gemma3 ↔ qwen2.5 = déchargement + rechargement (~30–60 s). nomic-embed-text partage le GPU → indexation RAG pendant E4/E5 = conflit avec qwen2.5.

## Semaine 0 : Setup (RTX 3060 12 Go)

L'opérateur suit la checklist 4.1 sur une machine équipée uniquement d'une RTX 3060 12 Go :

1. Ollama + modèles (qwen2.5-coder:7b, gemma3:12b-it-q4_K_M, nomic-embed-text) ✅
2. `OLLAMA_KEEP_ALIVE` non configuré par défaut → à documenter
3. `.agile-env` : **AGILE_DEFER_INDEX=true** (obligatoire pour RTX 3060)
4. **AGILE_RAG_FILE_WATCHER=false** (recommandé : éviter indexation temps réel pendant E4/E5)

### CC1 — OLLAMA_KEEP_ALIVE non documenté dans checklist 4.1

La stratégie GPU (III.8-J) mentionne `OLLAMA_KEEP_ALIVE` pour le modèle prioritaire, mais la checklist 4.1 n'inclut pas cette configuration. Un opérateur RTX 3060 qui n'a pas configuré `OLLAMA_KEEP_ALIVE=qwen2.5-coder:7b` pendant E4 subit des rechargements inutiles si Ollama décharge le modèle après inactivité.

**Correction** : Ajouter dans checklist 4.1 (ou section dédiée RTX 3060) : `export OLLAMA_KEEP_ALIVE=qwen2.5-coder:7b` pour sessions E4/E5 longues. Optionnel pour Phase 0 / E2 : `OLLAMA_KEEP_ALIVE=gemma3:12b-it-q4_K_M`.

## Semaines 3–5 : E4 Sprint 1 — Point critique GPU

R-3 produit Sprint Backlog. R-4/R-5 exécutent (qwen2.5-coder:7b chargé).

**AGILE_DEFER_INDEX=true** : chaque commit R-5 → hook écrit dans `pending_index.log` au lieu de lancer index_rag. **Aucun conflit nomic-embed-text ↔ qwen2.5**. ✅

**CC2 — Continue.dev pendant E4 : comportement indéfini**

Si Nghia garde Continue.dev ouvert (RAG + autocomplétion) pendant E4 :
- Continue.dev utilise son propre RAG (LanceDB/voyage-code) et peut solliciter un modèle local.
- Si Continue.dev = qwen2.5 (même que R-4) : pas de conflit de chargement, mais partage du GPU.
- Si Continue.dev = gemma3 : Ollama doit switcher ; R-4 ralentit ou timeout.

Le spec ne précise pas : "Pendant E4/E5, désactiver Continue.dev ou le configurer sur le même modèle que le graphe (qwen2.5-coder)".

**Correction** : Documenter dans III.8-J ou section "Recommandations RTX 3060" : pendant E4/E5, utiliser Continue.dev uniquement avec qwen2.5-coder (aligné sur R-4) ou désactiver l'autocomplétion IA si modèle différent.

## Conclusion Simulation 014

Le flux Agile est **valide sur RTX 3060 12 Go** à condition de :
1. **AGILE_DEFER_INDEX=true** systématique
2. **AGILE_RAG_FILE_WATCHER=false** (ou désactivation pendant E4/E5)
3. Pas de concurrence Continue.dev (modèle différent) pendant Phase 0, E2, E4, E5

Trois améliorations documentaires (CC1–CC3) identifiées. Aucun blocage fonctionnel. La stratégie GPU (III.8-J) est adaptée au matériel ; les recommandations CC1–CC2 renforceraient l'expérience opérateur sur configuration 12 Go.
