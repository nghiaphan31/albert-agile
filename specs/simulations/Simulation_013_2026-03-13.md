### BB2 — W5 vs Y4 : incohérence start_phase hotfix

La section **W5 (III.14)** dit : `--start-phase E4`. La section **Y4 (III.16)** dit : `start_phase: "HOTFIX"`. Ces deux descriptions coexistent et se contredisent.

- W5 a été écrit avant Y4. Y4 a introduit `start_phase HOTFIX` pour résoudre le problème du Sprint Backlog vide (Y4). Mais W5 n'a jamais été mis à jour.
- Un opérateur lisant W5 utiliserait `--start-phase E4` et obtiendrait un Sprint Backlog vide → crash.

**Correction** : Mettre à jour W5 pour utiliser `--start-phase HOTFIX` (conforme à Y4). Supprimer la mention `--start-phase E4` dans W5.

### BB3 — Hotfix merge develop→CI non défini

W5 dit : « R-5 merge hotfix→main + hotfix→develop (deux merges séquentiels). » Mais pour le merge hotfix→develop :
- Faut-il une PR ? Un CI ?
- Si `github_repo` est configuré, le merge hotfix→develop devrait-il passer par `gh pr create --base develop --head hotfix/...` et un CI ?
- Ou est-ce un merge direct (fast-forward) car le hotfix a déjà été validé sur main ?

**Correction** : Pour le merge hotfix→develop : merge direct `git merge hotfix/...` sans PR ni CI supplémentaire (le code a déjà été validé par CI 1 sur hotfix→main). R-5 exécute `git checkout develop && git merge --no-ff hotfix/... && git push`. Documenter explicitement.

### BB4 — auto_next_sprint=true : timing de la boucle E3 non précisé par rapport à phase 2

S4 (III.10) dit : si `auto_next_sprint=true`, boucle vers E3. Mais sprint_complete a deux phases (Z1). La boucle vers E3 doit se produire **après la phase 2** (merge_to_main complété, sprint_number incrémenté). Si la boucle se déclenche après la phase 1 (avant CI 2), le sprint N+1 démarrerait avant que le sprint N soit réellement intégré dans main.

**Correction** : Préciser dans S4 que si `auto_next_sprint=true`, la boucle vers E3 se déclenche uniquement après la clôture de **phase 2** (merge_to_main vert, sprint_number incrémenté). En mode dégradé (github_repo absent), la boucle se déclenche après la phase 1 (incrément immédiat).
