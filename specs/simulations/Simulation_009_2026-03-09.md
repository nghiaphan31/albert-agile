## Semaine 3–5 : E2, H2 → DoD + Architecture, GitHub Actions

### X1 — Incohérence PR target entre V4 et W4 dans le spec

III.13 (V4) dit : « GitHub Actions déclenché sur PR develop→main. »  
III.14 (W4) dit : « GitHub Actions tourne sur la PR feature→develop. »  
III.6 (H4) dit : « GitHub Actions tourne sur la PR feature→develop. »

Deux PR, deux CI. Lesquelles sont définitives ? Est-ce que les deux courent en même temps ? La spec doit préciser explicitement :
- CI 1 (obligatoire, pre-H4) : GitHub Actions sur PR **feature/{project_id}-sprint-{NN} → develop**.
- CI 2 (obligatoire, post-H4, avant merge main) : GitHub Actions sur PR **develop → main**.
- Le graphe (H4) attend CI 1 vert. Merge develop→main = action R-5 post-E6 (hors graphe ou nœud dédié).

## Semaine 5–7 : E4 sprint 1

### X2 — Timing de git push par R-5 dans E4 non défini

R-5 gère les commits Git en E4 (nœud LangGraph). W4 dit « R-5 pousse la feature branch avant H4 ». Mais quand exactement ?
- Option A : R-5 pousse après *chaque* commit de ticket → nombreux pushes, GitHub Actions déclenché en continu (coûteux).
- Option B : R-5 pousse une seule fois à la fin de E4 (tous les tickets commités) → un seul trigger GitHub Actions.
- Option C : R-5 pousse au début de E5 (avant que R-6 lance les tests locaux).

**Manque** : Le spec doit spécifier que R-5 effectue un **push unique en fin de E4** (après le dernier commit), juste avant que le graphe entre dans E5. Ce push ouvre la PR feature→develop sur GitHub. GitHub Actions CI 1 et R-6 CI local tournent en parallèle. H4 attend les deux.

## Semaine 7–9 : E5, GitHub Actions polling

### X3 — Comment le graphe LangGraph attend-il GitHub Actions ?

W4 dit que H4 ne se déclenche que si GitHub Actions est vert. Mais GitHub Actions peut prendre plusieurs minutes (5–20 min). Le nœud E5 (R-6) doit attendre ce résultat. Comment ?

Options non documentées :
- **Polling** : R-6 appelle l'API GitHub (`GET /repos/{owner}/{repo}/actions/runs`) toutes les N secondes. Timeout configurable.
- **Webhook → interrupt** : GitHub Actions envoie un webhook à LangServe quand le run se termine. LangServe déclenche un `Command(resume={"ci_status": "success|failure"})`.
- **Manual check** : L'opérateur vérifie GitHub Actions et résout l'interrupt manuellement.

**Manque** : Choisir une approche et la documenter. Recommandation : polling par le nœud E5 (R-6) via `run_shell("gh run watch --repo {repo} --exit-status")` (GitHub CLI). Timeout : `GITHUB_ACTIONS_TIMEOUT=600` secondes (10 min défaut). Si timeout : interrupt H5 avec `reason="github_actions_timeout"`.

## Semaine 11–12 : load_context failure

### X4 — Gestion d'erreur si BaseStore inaccessible dans load_context

load_context lit adr_counter, sprint_number, DoD depuis BaseStore. Si BaseStore (PostgresStore ou Chroma) est inaccessible (service down, fichier SQLite corrompu) :
- Option A : load_context échoue → le thread ne démarre pas. Message d'erreur clair.
- Option B : load_context utilise les valeurs par défaut (0, 1, None) avec un log WARNING. Le thread démarre en mode dégradé.

**Manque** : Option B recommandée pour les projets en développement (ne pas bloquer le développeur). Mais Option A obligatoire en production (éviter des collisions de compteurs). Variable `AGILE_BASESTORE_STRICT=false` (défaut) : si true, load_context échoue si BaseStore inaccessible. Si false : valeurs par défaut + WARNING.
