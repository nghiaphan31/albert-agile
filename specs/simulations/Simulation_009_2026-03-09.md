## Semaine 3–5 : E2 (architecture), H2 (validation Architecture + DoD) (validation Architecture + DoD (Definition of Done)) → DoD (Definition of Done) + Architecture, GitHub Actions

### X1 — Incohérence PR target entre V4 et W4 dans le spec

III.13 (V4) dit : « GitHub Actions déclenché sur PR develop→main. »  
III.14 (W4) dit : « GitHub Actions tourne sur la PR feature→develop. »  
III.6 (H4 (Sprint Review)) dit : « GitHub Actions tourne sur la PR feature→develop. »

Deux PR, deux CI. Lesquelles sont définitives ? Est-ce que les deux courent en même temps ? La spec doit préciser explicitement :
- CI 1 (feature→develop) (obligatoire, pre-H4 (Sprint Review)) : GitHub Actions sur PR **feature/{project_id}-sprint-{NN} → develop**.
- CI 2 (develop→main) (obligatoire, post-H4 (Sprint Review), avant merge main) : GitHub Actions sur PR **develop → main**.
- Le graphe (H4 (Sprint Review)) attend CI 1 (feature→develop) vert. Merge develop→main = action R-5 (Albert Release Manager) post-E6 (clôture sprint) (hors graphe ou nœud dédié).

## Semaine 5–7 : E4 (exécution code) sprint 1

### X2 — Timing de git push par R-5 (Albert Release Manager) dans E4 (exécution code) non défini

R-5 (Albert Release Manager) gère les commits Git en E4 (exécution code) (nœud LangGraph). W4 dit « R-5 (Albert Release Manager) pousse la feature branch avant H4 (Sprint Review) ». Mais quand exactement ?
- Option A : R-5 (Albert Release Manager) pousse après *chaque* commit de ticket → nombreux pushes, GitHub Actions déclenché en continu (coûteux).
- Option B : R-5 (Albert Release Manager) pousse une seule fois à la fin de E4 (exécution code) (tous les tickets commités) → un seul trigger GitHub Actions.
- Option C : R-5 (Albert Release Manager) pousse au début de E5 (tests CI) (avant que R-6 (Albert QA & DevOps) lance les tests locaux).

**Manque** : Le spec doit spécifier que R-5 (Albert Release Manager) effectue un **push unique en fin de E4 (exécution code)** (après le dernier commit), juste avant que le graphe entre dans E5 (tests CI). Ce push ouvre la PR feature→develop sur GitHub. GitHub Actions CI 1 (feature→develop) et R-6 (Albert QA & DevOps) CI local tournent en parallèle. H4 (Sprint Review) attend les deux.

## Semaine 7–9 : E5 (tests CI), GitHub Actions polling

### X3 — Comment le graphe LangGraph attend-il GitHub Actions ?

W4 dit que H4 (Sprint Review) ne se déclenche que si GitHub Actions est vert. Mais GitHub Actions peut prendre plusieurs minutes (5–20 min). Le nœud E5 (tests CI) (R-6 (Albert QA & DevOps)) doit attendre ce résultat. Comment ?

Options non documentées :
- **Polling** : R-6 (Albert QA & DevOps) appelle l'API GitHub (`GET /repos/{owner}/{repo}/actions/runs`) toutes les N secondes. Timeout configurable.
- **Webhook → interrupt** : GitHub Actions envoie un webhook à LangServe quand le run se termine. LangServe déclenche un `Command(resume={"ci_status": "success|failure"})`.
- **Manual check** : L'opérateur vérifie GitHub Actions et résout l'interrupt manuellement.

**Manque** : Choisir une approche et la documenter. Recommandation : polling par le nœud E5 (tests CI) (R-6 (Albert QA & DevOps)) via `run_shell("gh run watch --repo {repo} --exit-status")` (GitHub CLI). Timeout : `GITHUB_ACTIONS_TIMEOUT=600` secondes (10 min défaut). Si timeout : interrupt H5 (approbation escalade API payante) avec `reason="github_actions_timeout"`.

## Semaine 11–12 : load_context failure

### X4 — Gestion d'erreur si BaseStore (mémoire long terme) inaccessible dans load_context

load_context lit adr_counter, sprint_number, DoD (Definition of Done) depuis BaseStore (mémoire long terme). Si BaseStore (mémoire long terme) (PostgresStore ou Chroma) est inaccessible (service down, fichier SQLite corrompu) :
- Option A : load_context échoue → le thread ne démarre pas. Message d'erreur clair.
- Option B : load_context utilise les valeurs par défaut (0, 1, None) avec un log WARNING. Le thread démarre en mode dégradé.

**Manque** : Option B recommandée pour les projets en développement (ne pas bloquer le développeur). Mais Option A obligatoire en production (éviter des collisions de compteurs). Variable `AGILE_BASESTORE_STRICT=false` (défaut) : si true, load_context échoue si BaseStore (mémoire long terme) inaccessible. Si false : valeurs par défaut + WARNING.
