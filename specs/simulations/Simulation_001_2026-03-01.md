## Scénario de simulation

**Projet fictif** : Application web de gestion de tâches (kanban) en React + API Python FastAPI. Un seul développeur (Nghia) cumulant les rôles humains R-1 (Nghia Product Owner) et R-7 (Nghia Stakeholder). Sprints de 2 semaines.

## SEMAINE 1 : Phase 0 + Initialisation + E2 (architecture)

### J1–J2 : Phase 0 — Discovery

| Jour | Événement | Ce qui se passe concrètement |
|------|-----------|------------------------------|
| J1 | R-1 (Nghia Product Owner) lance Phase 0 | Nghia ouvre VS Code, démarre une conversation avec R-0 (Albert Business Analyst) (via Continue.dev ou le graphe LangGraph). Il décrit l'idée : "app kanban pour petites équipes". |
| J1 | R-0 (Albert Business Analyst) itère | R-0 (Albert Business Analyst) (gemma3:12b local) propose des personas, des hypothèses. Nghia challenge. 3–4 allers-retours. Tout en local → 0 €. |
| J2 | Cristallisation | R-0 (Albert Business Analyst) produit un Gros Ticket (Epic) structuré. **H1 (validation Gros Ticket)** : Nghia valide. Le graphe LangGraph est en `interrupt()` ; Nghia doit explicitement envoyer `Command(resume="approved")` pour continuer. |

**Problème potentiel P1** : Aucune interface n'est spécifiée pour consommer les `interrupt()`. Le document dit "CLI, script, ou future UI". En pratique : Nghia doit soit écrire un script Python qui affiche le payload et attend une entrée, soit utiliser le playground LangServe. **Manque** : Procédure opérationnelle (runbook) pour gérer les interrupts.

### J3 : E1 (idéation) — Initialisation

| Événement | Détail |
|-----------|--------|
| R-1 (Nghia Product Owner) injecte le Gros Ticket | Nghia copie-colle le Gros Ticket dans `Besoins - Product Backlog.md`. Le fichier est créé ou mis à jour. |
| Continue.dev indexe | Continue.dev indexe la codebase. Mais au J3, la codebase est vide (nouveau projet). Index vide. |
| RAG (recherche sémantique) Chroma | Le graphe LangGraph doit indexer le Backlog dans Chroma. **Qui déclenche l'indexation ?** Le document dit "après chaque commit significatif ou en fin de sprint". À J3 : premier commit (Backlog). L'indexation n'est pas automatiquement branchée sur le flux E1 (idéation). |

**Problème potentiel P2** : **Pas de pipeline d'indexation RAG (recherche sémantique) déclenché** par les événements du graphe. Le document décrit quoi indexer mais pas *quand ni comment* l'indexation est lancée. Risque : RAG (recherche sémantique) vide ou obsolète pendant les premières semaines.

### J4–J5 : E2 (architecture) — Backlog Refinement (R-2 (Albert System Architect))

| Événement | Détail |
|-----------|--------|
| R-2 (Albert System Architect) lit le Backlog | Le nœud R-2 (Albert System Architect) du graphe reçoit le Backlog. Il interroge le RAG (recherche sémantique) pour "architecture existing, stack technique". RAG (recherche sémantique) quasi vide → peu de contexte. |
| R-2 (Albert System Architect) produit Architecture.md | R-2 (Albert System Architect) (gemma3:12b) génère une architecture. Pour un projet neuf, le modèle local suffit souvent. Si la complexité augmente → escalade N1 (cloud gratuit) (Gemini). Limites : 250–500 RPD. Si Nghia a déjà utilisé Gemini pour autre chose dans la journée, risque **HTTP 429**. |
| H2 (validation Architecture + DoD) (validation Architecture + DoD (Definition of Done)) — Validation architecture | R-7 (Nghia Stakeholder) (Nghia) doit valider. Nghia est en déplacement : le graphe reste bloqué sur `interrupt()` pendant 2 jours. |

**Problème potentiel P3** : **Interrupts bloquants en absence de l'humain**. Si Nghia est indisponible (congés, urgence), tout le pipeline s'arrête. Aucune notion de délai maximum, d'escalade ou de reprise différée documentée.

**Problème potentiel P4** : **Conflit de ressources GPU**. Pendant E2 (architecture), gemma3:12b tourne sur Ollama. Si Nghia ouvre parallèlement Continue.dev pour une autre tâche, les deux peuvent solliciter le GPU. Ollama charge un seul modèle à la fois par défaut ; le switching peut ralentir ou provoquer des timeouts.

## SEMAINE 2–3 : Premier sprint (E3 (Sprint Backlog) → E6 (clôture sprint))

### J6–J7 : E3 (Sprint Backlog) — Sprint Planning (R-3 (Albert Scrum Master))

| Événement | Détail |
|-----------|--------|
| R-3 (Albert Scrum Master) découpe en tickets | R-3 (Albert Scrum Master) (qwen2.5-coder:7b) produit le Sprint Backlog. 8 tickets. Le modèle local gère bien ce type de tâche. |
| H3 (validation Sprint Backlog) — Validation découpage | Nghia valide. Le graphe reprend. |
| Checkpointer | État sauvegardé avec `thread_id=sprint-2026-01`. |

### J8–J12 : E4 (exécution code) — Sprint Execution (R-4 (Albert Dev Team), R-5 (Albert Release Manager))

| Jour | Événement | Détail |
|------|-----------|--------|
| J8 | R-4 (Albert Dev Team) prend le ticket 1 | Roo Code (dans VS Code) ou le nœud R-4 (Albert Dev Team) du graphe ? **Incohérence** : Le document dit "Roo Code + Cascade" pour E4 (exécution code), mais le graphe LangGraph a un nœud R-4 (Albert Dev Team). Roo Code est une extension VS Code ; LangGraph tourne en Python. Comment R-4 (Albert Dev Team) (graphe) et Roo Code interagissent-ils ? |
| J8 | R-4 (Albert Dev Team) génère du code | Si R-4 (Albert Dev Team) = nœud LangGraph : il appelle l'LLM, produit du code, l'écrit dans des fichiers. Mais LangGraph ne manipule pas le système de fichiers par défaut — il faut des *tools* (write_file, read_file). Ces tools ne sont pas spécifiés. |
| J9 | R-5 (Albert Release Manager) gère Git | R-5 (Albert Release Manager) doit créer une branche, committer. LangGraph peut appeler des tools shell (`git checkout -b feature/xxx`). Mais : **qui exécute** ? Le processus Python du graphe doit avoir accès au dépôt Git. Contexte : environnement de développement, chemins absolus. |
| J10 | R-4 (Albert Dev Team) enchaîne tickets 2–3 | R-4 (Albert Dev Team) interroge le RAG (recherche sémantique) pour "code existant similar". La codebase grossit ; le RAG (recherche sémantique) doit être ré-indexé. **Quand ?** Pas de hook post-commit défini. |
| J11 | Conflit de merge | R-5 (Albert Release Manager) détecte un conflit (simulation : modification manuelle par Nghia en parallèle). R-5 (Albert Release Manager) (qwen2.5-coder) doit résoudre. Modèle 7B peut échouer sur des conflits complexes. Escalade N1 (cloud gratuit) ou N2 (cloud payant). |
| J12 | R-4 (Albert Dev Team) termine ticket 8 | Sprint Backlog épuisé. R-4 (Albert Dev Team) signale la fin. |

**Problème potentiel P5** : **Articulation LangGraph ↔ Roo Code non définie**. Deux exécuteurs potentiels pour R-4 (Albert Dev Team) : (a) Roo Code dans VS Code, (b) nœud LangGraph en Python. Le document ne précise pas qui pilote quoi. Si Roo Code : comment le graphe sait-il que R-4 (Albert Dev Team) a terminé ? Si LangGraph : comment le code est-il édité dans VS Code ?

**Problème potentiel P6** : **Pas de pipeline d'indexation incrémentale**. Le RAG (recherche sémantique) doit être mis à jour après chaque commit ou à intervalle régulier. Aucun mécanisme (Git hook, cron, ou étape du graphe) n'est décrit.

**Problème potentiel P7** : **R-5 (Albert Release Manager) et conflits Git**. La résolution automatique de conflits par LLM est risquée. En cas d'échec répété, le Self-Healing ne s'applique pas à R-5 (Albert Release Manager) (il s'applique à R-6 (Albert QA & DevOps)→R-4 (Albert Dev Team)). Un conflit non résolu bloque tout le sprint.

### J13–J14 : E5 (tests CI) — CI/CD (R-6 (Albert QA & DevOps))

| Événement | Détail |
|-----------|--------|
| R-6 (Albert QA & DevOps) lance les tests | Docker, Jest/Pytest. Les tests s'exécutent. 2 tests échouent. |
| R-6 (Albert QA & DevOps) analyse les logs | R-6 (Albert QA & DevOps) (Cascade) reçoit les logs. Interroge le RAG (recherche sémantique) ? Ou envoie les logs à l'LLM ? Les logs peuvent être longs (plusieurs KB). Contexte limité à 128K tokens, mais accumulation possible. |
| Self-Healing | R-6 (Albert QA & DevOps) renvoie vers R-4 (Albert Dev Team). R-4 (Albert Dev Team) corrige. Nouveau cycle. 2ème tentative : 1 test encore en échec. 3ème tentative : tous les tests passent. |
| Verdict | R-6 (Albert QA & DevOps) valide. Passage à E6 (clôture sprint). |

**Problème potentiel P8** : **Seuil max Self-Healing**. Le document dit "seuil max d'itérations (ex. 3)" mais ce n'est pas une règle stricte. Si 10 itérations échouent, le graphe peut boucler indéfiniment ou jusqu'à épuisement des ressources (API, temps).

### J14 : E6 (clôture sprint) — Sprint Review (R-7 (Nghia Stakeholder))

| Événement | Détail |
|-----------|--------|
| H4 (Sprint Review) — Go/No-Go | Nghia (R-7 (Nghia Stakeholder)) valide l'incrément. Go. |
| Retrofit | De nouveaux besoins émergent (Nghia a testé l'app). Il met à jour le Backlog manuellement. Le graphe ne gère pas automatiquement le Retrofit : c'est une action humaine sur le fichier. |

**Problème potentiel P9** : **Retrofit non automatisé**. Le document dit "R-1 (Nghia Product Owner) met à jour le Backlog". En pratique, Nghia édite le fichier. Le RAG (recherche sémantique) Chroma doit être ré-indexé pour prendre en compte le nouveau Backlog. Aucun trigger automatique.

## SEMAINE 4–5 : Deuxième sprint — accumulation

### Nouveaux tickets, codebase qui grandit

| Événement | Observation |
|-----------|-------------|
| RAG (recherche sémantique) | La codebase a ~50 fichiers. Chroma contient des chunks. Mais si l'indexation n'a jamais été lancée correctement, ou si elle est lancée une seule fois en début de projet, le RAG (recherche sémantique) est partiellement obsolète. |
| Checkpointer | `thread_id=sprint-2026-02`. Nouveau thread. L'historique du sprint 1 est dans le checkpointer du thread 1, pas dans le thread 2. La mémoire long terme (BaseStore (mémoire long terme)) devrait contenir des résumés. **Est-ce que les nœuds écrivent effectivement dans le BaseStore (mémoire long terme) ?** Le document décrit l'architecture mais pas le flux d'écriture. |
| Limites Gemini | En 4 semaines, Nghia a fait Phase 0 + 2 sprints. Plusieurs dizaines d'appels à Gemini. Si une journée concentre beaucoup d'activité (E2 (architecture) + E3 (Sprint Backlog) + plusieurs tickets), 250–500 RPD peuvent être atteints. HTTP 429. |
| LangSmith | 5000 traces/mois. Chaque invocation de graphe, chaque appel LLM = trace. En 1 mois, facile d'atteindre 2000–3000 traces. Après 2–3 mois, dépassement possible. |

**Problème potentiel P10** : **LangSmith free tier saturé**. 5000 traces/mois, rétention 14 jours. Si dépassement : plus de traçage, débogage difficile. Pas de stratégie de réduction (ex. tracer uniquement les échecs, ou échantillonner).

**Problème potentiel P11** : **BaseStore (mémoire long terme) jamais alimenté**. Le document décrit le BaseStore (mémoire long terme) comme mémoire long terme, mais ne spécifie pas quels nœuds y écrivent, à quel moment, ni sous quel format. Risque : BaseStore (mémoire long terme) vide, pas de bénéfice cross-session.

## SEMAINE 6+ : Problèmes structurels

### Dérive et dette technique

| Problème | Description |
|----------|-------------|
| **Architecture.md vs réalité** | L'architecture évolue (choix techniques faits par R-4 (Albert Dev Team) ou Nghia en cours de route). Architecture.md n'est pas automatiquement synchronisé. Les agents continuent de s'appuyer sur une version obsolète. |
| **ADRs** | Les ADRs sont obligatoires pour les décisions architecturales. Mais quand R-2 (Albert System Architect) ou R-4 (Albert Dev Team) prend une micro-décision (ex. choix d'une librairie), qui rédige l'ADR (Architecture Decision Record) ? Pas d'automatisation. |
| **DoD (Definition of Done) évolutive** | La DoD (Definition of Done) peut changer entre sprints. R-6 (Albert QA & DevOps) applique "la" DoD (Definition of Done). Quelle version ? Celle du fichier à l'instant T. Si le fichier est modifié pendant un sprint, incohérence. |
| **Multi-projets** | Si Nghia lance un 2ème projet (autre repo), le graphe, Chroma, le checkpointer doivent gérer plusieurs projets. `thread_id` pourrait inclure le projet. Chroma : une collection par projet ? Pas spécifié. |

### Ressources matérielles

| Contrainte | Impact |
|------------|--------|
| **12 Go VRAM** | qwen2.5-coder:7b (~8–10 Go) + gemma3:12b (~8 Go) ne tournent pas en parallèle. Ollama charge un modèle à la fois. Si le graphe utilise gemma3 et Roo Code utilise qwen2.5-coder, il y a swapping. Plus lent. |
| **nomic-embed-text** | Les embeddings consomment aussi du GPU/CPU. Pendant une indexation Chroma massive, Ollama peut être sollicité pour nomic-embed-text. Conflit avec les LLM. |
| **Disque** | Checkpointer Sqlite, Chroma, dépôts Git, logs LangSmith (si export local). Croissance continue. Pas de politique de purge. |

**Problème potentiel P12** : **Pas de politique de rétention et de purge**. Les checkpoints s'accumulent. Chroma grandit. Les logs aussi. Après 6 mois, plusieurs Go de données. Aucune règle pour archiver ou supprimer les threads anciens.

## Conclusion de la simulation

L'écosystème est cohérent sur le papier mais souffre de **manques opérationnels** :

1. **Interfaces et runbooks** : Les interrupts (H1 (validation Gros Ticket)–H5 (approbation escalade API payante)) nécessitent une interface claire ; sans elle, l'humain ne peut pas interagir efficacement.
2. **Articulation des composants** : LangGraph, Roo Code, Continue.dev — les frontières et les points de passage ne sont pas assez précis.
3. **Pipelines manquants** : Indexation RAG (recherche sémantique), alimentation BaseStore (mémoire long terme), sync des artefacts.
4. **Limites et seuils** : Pas de gestion explicite des quotas (Gemini, LangSmith), des boucles infinies, ni de la croissance des données.
5. **Résilience** : Conflits Git, indisponibilité humaine, conflits GPU — peu de procédures de fallback.

**Référence pour itérations** : Les corrections issues de cette simulation sont intégrées dans le document de spécification principal (section IX et sections modifiées). La simulation 002 et suivantes devront vérifier que ces corrections sont effectives.
