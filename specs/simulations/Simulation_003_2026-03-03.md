*Cette simulation valide l'écosystème complet sur un projet différent (API vs kanban), en conditions multi-projets et sur une durée longue (10 semaines). Objectif : identifier les problèmes résiduels, les cas limites et les améliorations.*

## Scénario de simulation

**Projet principal** : API REST météo (Python FastAPI, Pydantic, endpoints CRUD, base SQLite). Un seul développeur (Nghia = R-1 + R-7), sprints de 2 semaines.

**Projet en parallèle** : Kanban (React + FastAPI) — sprint 3 en cours au démarrage. Même graphe, `projects.json` configuré, deux `thread_id` actifs.

**Corrections supposées implémentées** : `handle_interrupt.py --thread-id`, `index_rag.py` (orchestration, `--project-root`, `--project-id`, `--sources`), nœuds post_h1/post-H2/post-H4, ordre BaseStore → index_rag → transition, état avec `project_root`/`project_id`, `projects.json`, résilience index_rag (try/except par fichier).

**Problème potentiel S5** : **Rapport index_rag non consommé** : Le script produit un rapport (fichiers indexés, en erreur). Qui le consulte ? Si le hook Git lance le script en arrière-plan, le rapport peut être perdu (stdout non capturé). Pas de mécanisme d'alerte si trop de fichiers en erreur (ex. > 10 %). Option `--strict` : exit 1 si aucun succès — mais le hook peut ignorer le code de sortie.

### J19 : Journée de stress — HTTP 429

| Événement | Détail |
|-----------|--------|
| Contexte | Nghia a utilisé Gemini pour une démo le matin. Limite RPD approchée. |
| R-4 | Ticket complexe. qwen2.5-coder échoue (qualité). Escalade N1 (Gemini). **HTTP 429** (Rate Limit Exceeded). |
| Cascade | Le graphe tente N1, reçoit 429. Backoff exponentiel (documenté ?). Après retries, échec. Escalade N2 (Sonnet). |
| H5 | `interrupt("H5", {"reason":"escalade N2","estimate":...})`. Nghia approuve. Sonnet exécute. Ticket résolu. ✅ |

**Problème potentiel S6** : **Backoff sur 429** : Le document mentionne "Exponential Backoff obligatoire" (Annexe A) mais pas dans les sections techniques III. Les connecteurs LangChain (Google, Anthropic) ont-ils un backoff natif ? Si oui, combien de retries ? Si 429 persiste après 3 retries, le graphe peut-il distinguer "429 temporaire" vs "quota épuisé" et adapter (H5 pour approbation N2 vs pause et retry plus tard) ? Non spécifié.

### J20 : index_rag --strict

| Événement | Détail |
|-----------|--------|
| Contexte | Après un refactor massif, plusieurs fichiers ont des erreurs de syntaxe temporaires. |
| index_rag | `--strict` : si aucun fichier indexé avec succès → exit 1. Le hook Git reçoit exit 1. Git considère le hook en échec ? **Non** : les hooks post-commit ne bloquent pas le commit (déjà fait). Le hook post-commit est exécuté après le commit. Un exit 1 de index_rag ne peut pas annuler le commit. Donc `--strict` sert à alerter (logs, monitoring) mais pas à bloquer. |
| Impact | RAG potentiellement vide ou partiel. R-4 peut manquer de contexte. Pas d'alerte automatique à Nghia. |

**Problème potentiel S7** : **Alerte sur échec index_rag** : Si index_rag échoue (--strict) ou si trop de fichiers en erreur, qui est notifié ? Un script wrapper pourrait envoyer un email ou écrire dans un fichier `logs/index_errors.log` consultable. Non documenté.

## Conclusion Simulation 003

L'écosystème fonctionne en conditions **multi-projets et longue durée** lorsque toutes les corrections Sim 001 et 002 sont appliquées. Les principaux **manques identifiés** concernent :

1. **Bootstrap opérationnel** : Installation des hooks Git dans les nouveaux dépôts (S2).
2. **Flux cyclique** : Passage explicite du sprint N au sprint N+1 (S4).
3. **Observabilité** : Rapports index_rag, alertes, purge checkpoints détaillée (S5, S7, S8).
4. **Résilience API** : Backoff 429 et gestion quota (S6).
5. **Maintenance long terme** : Archivage Chroma, sync artefacts, génération docs (S10, S11, S12).
6. **Audit** : Rétention traces LangSmith (S9).

**Statut** : Toutes les corrections S1–S12 ont été intégrées dans le document de spécification (section III.10). Voir `Specifications Ecosysteme Agile Agent IA.md`.
