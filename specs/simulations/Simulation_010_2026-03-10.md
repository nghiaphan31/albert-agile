## Semaine 1 : Initialisation — branche develop manquante

### Y1 — Branche develop non créée : premier projet

V4/X1 disent : R-5 (Albert Release Manager) crée `feature/{project_id}-sprint-{NN}` depuis `develop`. Mais pour un nouveau projet, la branche `develop` n'existe pas. Qui la crée ? Quand ?

- Le hook `setup_project_hooks.sh` gère uniquement les hooks Git (post-commit). Il ne crée pas les branches.
- Le nœud E1 (idéation) (initialisation) crée le Backlog file, mais aucun nœud ne crée les branches Git de base.
- Si R-5 (Albert Release Manager) tente `git checkout -b feature/webhook-sprint-01 develop` sur un repo qui n't a que `main` : erreur.

**Manque** : `setup_project_hooks.sh` (ou un script `init_project_branches.sh`) doit créer la branche `develop` depuis `main` si elle n'existe pas : `git checkout -b develop main && git push -u origin develop`. Documenter comme étape obligatoire dans la checklist.

## Semaine 2–4 : Sprint 1, E4 (exécution code), push, CI 1 (feature→develop)

### Y2 — gh run watch : identification du run et authentification

X3 spécifie `gh run watch --repo {repo} --exit-status --interval 30`. Mais :
- `gh run watch` sans ID de run surveille le *dernier* run en cours sur la branche courante. Sur un repo actif, il pourrait surveiller un run d'une autre PR.
- La feature branch est poussée par R-5 (Albert Release Manager) (X2). La PR est-elle créée automatiquement, ou R-5 (Albert Release Manager) doit-il exécuter `gh pr create` ?
- Si la PR n'existe pas encore, GitHub Actions ne s'exécute que si le workflow est déclenché sur `push` (pas sur `pull_request`). La distinction est importante.
- `gh` nécessite une authentification : `gh auth login`. Non documentée dans la checklist.

**Manque** : (1) R-5 (Albert Release Manager) exécute `gh pr create --base develop --head feature/{project_id}-sprint-{NN}` après le push. (2) GitHub Actions workflow configuré sur `pull_request` (pas seulement `push`). (3) R-6 (Albert QA & DevOps) récupère le run ID via `gh pr checks {pr_number} --required --watch` plutôt que `gh run watch`. (4) Ajouter `gh auth login` à la checklist (étape unique, persistée dans ~/.config/gh).

## Semaine 5–6 : Post-E6 (clôture sprint), CI 2 (develop→main), merge develop→main

### Y3 — Nœud merge_to_main : défini ou manuel ?

X1 dit : « Post-E6 (clôture sprint) : R-5 (Albert Release Manager) ouvre PR develop→main → CI 2 (develop→main) (GitHub Actions, develop→main). Merge develop→main conditionné à CI 2 (develop→main) vert. Effectué par R-5 (Albert Release Manager) (nœud "merge_to_main" post-sprint_complete, ou action manuelle). »

La parenthèse « ou action manuelle » laisse une ambiguïté opérationnelle. Si c'est un nœud du graphe : le graphe n'est pas terminé après sprint_complete. Si c'est une action manuelle : l'opérateur doit se souvenir de le faire.

**Manque** : Clarifier et choisir une approche unique : le nœud sprint_complete s'assure que R-5 (Albert Release Manager) exécute `gh pr create --base main --head develop` + polling `gh pr checks --required --watch`. Si CI 2 (develop→main) échoue : interrupt H5 (approbation escalade API payante) avec `reason="ci2_github_actions_failure"`. R-4 (Albert Dev Team) corrige, nouveau cycle E5 (tests CI) (CI 1 (feature→develop)) puis nouveau push develop, CI 2 (develop→main) se relance. Merge develop→main exécuté par R-5 (Albert Release Manager) automatiquement après CI 2 (develop→main) vert (ou H4 (Sprint Review)' manuel pour validation finale).

## Semaine 7–8 : Hotfix

### Y4 — Hotfix avec --start-phase E4 (exécution code) : état incomplet

W5 dit : invocation `--start-phase E4 (exécution code) --thread-id project-hotfix-{date}`. Mais E4 (exécution code) utilise le Sprint Backlog (`state.sprint_backlog`). load_context charge les valeurs depuis BaseStore (mémoire long terme). Sprint Backlog = None pour un thread hotfix (jamais passé par E3 (Sprint Backlog)). Le nœud E4 (exécution code) tente de lire les tickets depuis state.sprint_backlog → crash ou liste vide.

**Manque** : Pour start_phase=E4 (exécution code), l'invocation doit passer le correctif à appliquer dans `configurable` : `{"hotfix_description": "Fix NPE in handler X"}`. Load_context crée un SprintBacklog synthétique d'un seul ticket (`HF-001: {hotfix_description}`). Ou : créer un `--start-phase HOTFIX (correctif urgent)` dédié qui by-passe E3 (Sprint Backlog) et injecte le Sprint Backlog directement.

---

## Semaine 9–10 : github_repo absent

### Y5 — Comportement si github_repo absent dans projects.json

X3 dit : `github_repo` est optionnel. Si absent, `gh run watch` ne peut pas fonctionner. Le spec ne définit pas ce qui se passe si `github_repo` n'est pas configuré.

**Manque** : Si `github_repo` absent (ou vide) dans projects.json : le nœud E5 (tests CI) ne lance pas `gh pr create` ni `gh pr checks`. CI 1 (feature→develop) est skippé (log WARNING : "github_repo non configuré, CI GitHub Actions skippé"). H4 (Sprint Review) est déclenché uniquement sur CI local (R-6 (Albert QA & DevOps)) vert. Ce mode dégradé doit être documenté explicitement : le projet n'a pas de CI GitHub Actions, seul le CI local est requis.
