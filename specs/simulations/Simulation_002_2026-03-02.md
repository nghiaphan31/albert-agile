**Problème résiduel R2** : **E1 (idéation) et indexation** : Le flux E1 (idéation) est une action humaine (injection du Backlog). Le graphe ne "fait" pas E1 (idéation). L'indexation "Fin E1 (idéation)" nécessite soit une étape graphe explicite "index_after_e1" invoquée après l'injection, soit un hook Git. Le hook Git (post-commit) est défini pour main/develop — à J3, le premier commit contient le Backlog. Le hook devrait lancer `index_rag.py`. ✅ Sous réserve que le hook soit configuré dans le dépôt du **projet** (kanban) et que `index_rag.py` soit exécutable depuis ce contexte (chemin vers le projet, variable `PROJECT_ROOT`).

**Problème résiduel R3** : **Où vit `index_rag.py` ?** Dans le projet orchestration (graphe) ou dans chaque projet (kanban) ? Si orchestration : le script doit recevoir en argument le chemin du projet pour indexer. Si projet : chaque projet doit avoir une copie du script ou un lien. Non spécifié.

| Événement | Détail |
|-----------|--------|

**Problème nouveau N1 (cloud gratuit)** : **Multi-projets non spécifié**. Le document évoque "une collection par projet" mais ne décrit pas la gestion des `project_id`, `project_root`, ni l'interface `handle_interrupt` pour plusieurs projets.

### Problème nouveau : erreur dans index_rag.py

| Événement | Détail |
|-----------|--------|
| Contexte | Un fichier du projet contient des caractères que nomic-embed-text ne gère pas bien. Ou le chunking produit un chunk trop long. |
| Résultat | `index_rag.py` plante. Le hook Git capture l'erreur mais le commit a déjà réussi. L'indexation est incomplète. |
| Impact | RAG (recherche sémantique) partiellement obsolète. R-4 (Albert Dev Team) pourrait ne pas retrouver du code pertinent. |

**Problème nouveau N2 (cloud payant)** : **Résilience de `index_rag.py`** : Pas de stratégie de retry, de gestion d'erreurs par fichier (skip et log), ni de validation des chunks. Un échec partiel peut passer inaperçu.

## Conclusion Simulation 002
