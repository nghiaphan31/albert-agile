### J3 : E1 — Initialisation

**Problème résiduel R2** : **E1 et indexation** : Le flux E1 est une action humaine (injection du Backlog). Le graphe ne "fait" pas E1. Le hook Git (post-commit) est défini pour main/develop — à J3, le premier commit contient le Backlog. Le hook devrait lancer `index_rag.py`. ✅ Sous réserve que le hook soit configuré dans le dépôt du **projet** (kanban) et que `index_rag.py` soit exécutable depuis ce contexte (chemin vers le projet, variable `PROJECT_ROOT`).

**Problème résiduel R3** : **Où vit `index_rag.py` ?** Dans le projet orchestration (graphe) ou dans chaque projet (kanban) ? Si orchestration : le script doit recevoir en argument le chemin du projet pour indexer. Si projet : chaque projet doit avoir une copie du script ou un lien. Non spécifié.

## SEMAINE 6–8 : Troisième sprint + stress

### Charge et limites

| Événement | Observation |
|-----------|-------------|
| LangSmith | 3 sprints + Phase 0 + E2, E3. Environ 400–500 invocations graphe. Chaque nœud = 1 ou plusieurs appels LLM. Estimation : 2000–3000 traces en 6 semaines. Sous la limite 5000. Si 4ème mois : risque de dépassement. Politique d'échantillonnage (0.2) ou `ERRORS_ONLY` peut être activée. |
| Gemini RPD | Journée chargée : E2 + E3 + plusieurs tickets R-4. 50–100 appels. Sous 250–500 RPD. Gérer les pics. |
| Chroma | Croissance linéaire. 100–200 fichiers, ~1000 chunks. Pas de problème. |
| Checkpoints | 3 threads (sprints). Purge à 90 jours : pas encore. |

### Problème nouveau : multi-projets

| Événement | Détail |
|-----------|--------|
| Contexte | Nghia lance un 2ème projet (API météo). Même machine, même graphe. |
| thread_id | `project-api-meteo-phase-0` ou `sprint-meteo-01`. |
| Chroma | Une collection par projet (doc III.7). Les scripts `index_rag.py` et le graphe doivent recevoir un paramètre `project_id` pour cibler la bonne collection. |
| project_root | Chaque projet a un chemin différent. Le graphe doit maintenir une table ou une config : `project_id → path`. |
| handle_interrupt | Si deux projets ont un interrupt en attente, `handle_interrupt.py` doit lister les threads et demander lequel traiter. |

**Problème nouveau N1** : **Multi-projets non spécifié**. Le document évoque "une collection par projet" mais ne décrit pas la gestion des `project_id`, `project_root`, ni l'interface `handle_interrupt` pour plusieurs projets.

### Problème nouveau : erreur dans index_rag.py

| Événement | Détail |
|-----------|--------|
| Contexte | Un fichier du projet contient des caractères que nomic-embed-text ne gère pas bien. Ou le chunking produit un chunk trop long. |
| Résultat | `index_rag.py` plante. Le hook Git capture l'erreur mais le commit a déjà réussi. L'indexation est incomplète. |
| Impact | RAG partiellement obsolète. R-4 pourrait ne pas retrouver du code pertinent. |

**Problème nouveau N2** : **Résilience de `index_rag.py`** : Pas de stratégie de retry, de gestion d'erreurs par fichier (skip et log), ni de validation des chunks. Un échec partiel peut passer inaperçu.
