## SEMAINE 1–2 : Séquence H1 → E1 → post_h1

### Problème T1 : Ordre des opérations H1 / E1 / post_h1

| Étape | Détail |
|-------|--------|
| H1 | R-0 produit le Gros Ticket. `interrupt("H1")`. Nghia valide. |
| E1 | R-1 doit injecter le Gros Ticket dans `Besoins - Product Backlog.md`. C'est une action manuelle. |
| post_h1 | Le nœud post_h1 s'exécute après `Command(resume="approved")` sur H1. Il appelle `index_rag.py --sources backlog`. |

**Ambiguïté** : Quand Nghia envoie `Command(resume="approved")`, le graphe exécute immédiatement post_h1. À ce moment, le fichier Backlog existe-t-il ? Si Nghia valide H1 *avant* d'avoir créé le Backlog (il valide le Gros Ticket à l'écran, puis va créer le fichier), post_h1 indexe un fichier inexistant ou un répertoire vide. **Ordre requis** : (1) valider H1, (2) créer/injecter le Backlog, (3) committer, (4) *puis* envoyer Command(resume) ? Ou le graphe doit-il avoir une étape "attente_injection" où il suspend jusqu'à détection du Backlog ? Non spécifié.

## SEMAINE 3–4 : setup_project_hooks et project_id

### Problème T2 : project_id dans le hook vs projects.json

Le hook post-commit appelle `index_rag.py --project-root $(pwd) --project-id $(basename $(pwd))`. Mais `projects.json` associe `"cli-backup": "/path/to/cli-backup"`. Si le dépôt s'appelle `cli_backup` (underscore), `basename $(pwd)` = `cli_backup`, pas `cli-backup`. La collection Chroma sera `cli_backup`, alors que le graphe utilise `project_id=cli-backup` (depuis projects.json). **Incohérence** : deux project_id pour le même projet.

### Problème T3 : setup_project_hooks — chemin ORCHESTRATION_ROOT figé

Le script `setup_project_hooks.sh` installe un hook qui contient un chemin absolu vers `index_rag.py` (ex. `/home/user/orchestration/scripts/index_rag.py`). Si Nghia déplace le projet orchestration (réorganisation des dossiers), tous les hooks des projets existants pointent vers un chemin invalide. **Manque** : utilisation d'une variable d'environnement `AGILE_ORCHESTRATION_ROOT` lue au moment de l'exécution du hook, plutôt qu'un chemin figé.

## SEMAINE 5–6 : auto_next_sprint et format thread_id

### Problème T4 : Emplacement de la config auto_next_sprint

S4 (III.10) précise que le nœud sprint_complete consulte `auto_next_sprint` pour décider de boucler vers E3. Où cette config vit-elle ? `projects.json` par projet ? Variable d'environnement globale ? Argument à chaque invocation ? Exemple souhaité : `{"kanban": {"path": "/path/kanban", "auto_next_sprint": true}}`. Non spécifié.

### Problème T5 : Format thread_id pour les sprints

Convention : `{project_id}-sprint-{n}`. Pour n=1 : `kanban-sprint-1` ou `kanban-sprint-01` ? Le tri alphabétique de `handle_interrupt` peut afficher `kanban-sprint-10` avant `kanban-sprint-2` si pas de padding. **Recommandation** : `sprint-01`, `sprint-02`, etc. (2 chiffres minimum).

## SEMAINE 7–10 : Cas limites opérationnels

### Problème T6 : build_docs et projet sans doc configurée

S12 : R-6 exécute build_docs (Sphinx/JSDoc) dans E5. Pour un projet Python minimal, Sphinx n'est peut-être pas configuré (pas de `docs/` ni `conf.py`). L'étape build_docs échoue. Le spec dit "option d'interrupt ou de skip selon la DoD". **Manque** : comportement par défaut si aucun `conf.py` ou `jsdoc.json` n'existe : skip silencieux ? échec ? Variable `BUILD_DOCS_OPTIONAL=true` ?

### Problème T7 : sync_artifacts — déclenchement

S11 : "Nœud optionnel ou tâche cron". Si optionnel, qui décide de l'activer ? Comment ? Pas de procédure.

### Problème T8 : Incohérence exit code handle_interrupt

III.8 indique "exit code 1" si thread sans interrupt. III.10 S1 indique "exit code 2". **Incohérence** à clarifier : 1 pour erreur générale, 2 pour thread invalide ? Ou normaliser sur 1.

## Synthèse des problèmes résiduels (Simulation 004)

| ID | Problème | Gravité | Recommandation |
|----|----------|---------|----------------|
| **T1** | Ordre H1 / E1 / post_h1 : Backlog peut ne pas exister quand post_h1 s'exécute | Critique | Spécifier : E1 (injection + commit) doit être faite AVANT Command(resume) sur H1. Ou : post_h1 suit un nœud "wait_for_backlog" qui vérifie l'existence du fichier. |
| **T2** | project_id hook (basename) ≠ project_id projects.json | Majeure | Le hook doit recevoir project_id explicitement. Option : fichier `.agile-project-id` à la racine du projet, ou config dans setup_project_hooks. |
| **T3** | ORCHESTRATION_ROOT figé dans le hook | Majeure | Hook lit `$AGILE_ORCHESTRATION_ROOT` au runtime. setup_project_hooks configure cette env dans le hook, ou le hook la récupère d'un fichier partagé. |
| **T4** | Config auto_next_sprint non localisée | Moyenne | Ajouter dans projects.json : `"<project_id>": {"path": "...", "auto_next_sprint": bool}`. Défaut : false. |
| **T5** | Format thread_id sprint (padding) | Faible | Convention : sprint-01, sprint-02 (2 chiffres min). Documenter dans III.9 N1. |
| **T6** | build_docs si pas de config Sphinx/JSDoc | Moyenne | Si pas de conf.py/jsdoc.json : skip avec log. Variable `BUILD_DOCS_REQUIRED=false` (défaut) pour optionnel. |
| **T7** | sync_artifacts déclenchement | Faible | Documenter : cron hebdo ou nœud optionnel invoqué manuellement. Variable `SYNC_ARTIFACTS_CRON="0 0 * * 0"` (dimanche minuit). |
| **T8** | Exit codes handle_interrupt | Faible | Normaliser : 0=succès, 1=erreur (thread invalide ou autre), 2=usage (arguments invalides). Documenter. |
